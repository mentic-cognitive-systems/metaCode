The WHAM Architecture: Building Better with MVC's Static and Dynamic Parts

Pattern Card — a practical, enforceable code-structuring meta-architecture for humans and machine agents.
Core idea: Static definitions + dynamic composition + pure transforms + declared effects

1) Vocabulary (Canon)

Component WHAM Role What it is (practical) Must / Must Not
WINDOW Static Models Declared context frames: screens/routes/panels/endpoints/DOM roots. Hosts event slots and renders from Identity snapshots. Must declare structure, render from state
Must not contain business logic or mutate state
ADAPTER Dynamic Models Event adapter/handler that bundles runtime context (Window + event + Identity ref) into an Intent. Must produce Intent
Must not do IO or commit state
IDENTITY Dynamic State Single source of truth for state. Versioned by entityId + revisionId. Commits deltas → new revision. Must own state + versioning
Must not know UI or routing
INTENT Dynamic Views Normalized “meaning” of an interaction: kind + capability + identityRef + payload + windowCtx + meta. Loggable/replayable. Must be stable/testable
Must not contain executable logic
INTERFACE Static Views Static capability map: capability → Action delegate/pipeline. Defines affordances/policy surfaces. Must be delegate map (data)
Must not hold state or route
ACTION Static Control Pure transform: (state, intent) → {delta, effects}. Deterministic. Composable pipelines. Must be pure + return effects
Must not perform IO
MANAGER Dynamic Control Runtime composer: Binder + Router + Workflow + Effect Runner. Binds events, routes intents, commits deltas, runs effects. Must be only committer/effect runner
Should be split internally to avoid “god object”
Non-negotiable contract #1
Adapter always returns an Intent (no IO, no mutation).
Non-negotiable contract #2
Action always returns { delta, effects[] } (pure).
Non-negotiable contract #3
Only Manager commits Identity + runs effects.
Non-negotiable contract #4
Interface is a capability map (delegate data), not just a type annotation.

2) Canonical Flow

Separate composition-time wiring from runtime execution.
COMPOSITION (binding)
Manager binds:
  Window.eventSlot → Adapter
Manager routes:
  Intent → Interface → Action delegate

RUNTIME (execution)
Event → Adapter → Intent → Manager.route()
  → Interface.delegate → Action(state,intent)
  → {Δ, Fx} → Manager.commit(Δ)
  → Manager.runEffects(Fx)
  → Window.render(identitySnapshot)

3) Folder Layout (Default)

/windows      (WINDOW: static models)
/adapters     (ADAPTER: dynamic models)
/identity     (IDENTITY: state + versioning)
/intents      (INTENT: schemas/helpers)
/interfaces   (INTERFACE: capability maps)
/actions      (ACTION: pure transforms)
/manager      (MANAGER: bind/route/workflow)
/effects      (effect executors)
/kinds        (policy + affordances)
/tests

4) Quick Placement Rules

  • Renders or declares structure → Window
  • Bundles event + context into meaning → Adapter → Intent
  • Stores/version state → Identity
  • Defines affordances as delegate map → Interface
  • Pure state transform → Action
  • Wires, routes, commits, runs effects → Manager

5) Mental Shifts

  • Events don’t “do work” — they produce Intent.
  • Interfaces are data (capability maps), not just types.
  • State updates are values (Δ), not mutations.
  • Effects are declared in Actions and executed by Manager.
  • Composition is a runtime responsibility (Manager), not scattered glue.
  • Identity is versioned (entityId + revisionId), enabling replay/audit.
